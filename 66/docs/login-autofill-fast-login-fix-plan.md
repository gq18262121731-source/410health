# 登录自动补全与慢登录修复方案

## 1. 问题背景

当前登录页存在两类直接影响联调效率的问题：

1. 打开 `http://localhost:5173/` 后，测试账号不会稳定自动回填，必须手动输入用户名。
2. 输入用户名和密码后，登录请求有时会长时间停留在“登录中...”，甚至最终报错。

这两个问题叠加后，会让人误以为“账号不对”或“前端坏了”，但实际根因主要集中在登录页兜底能力不足，以及登录相关请求缺少超时与轻量重试。

## 2. 根因分析

### 2.1 自动补全失效的根因

原逻辑依赖后端接口 `GET /api/v1/auth/mock-accounts` 成功返回后，才会：

- 初始化演示账号列表
- 选择默认账号
- 回填用户名和默认密码

因此只要首屏请求失败、后端刚启动尚未 ready、或者接口响应过慢，页面就会保持空白，不再自动补全。

### 2.2 登录缓慢的根因

原登录链路存在三个稳定性短板：

- 登录请求没有显式超时控制，`fetch` 在后端无响应时会长时间等待。
- 登录请求没有轻量重试，后端刚完成启动时的一次瞬时失败会直接暴露给用户。
- 会话恢复 `GET /api/v1/auth/me` 也没有超时，容易拖慢首屏。

### 2.3 运行现场风险

本次排查中还发现一个现场问题：`8000` 端口上曾经同时挂过多个 `uvicorn` 进程，且 `healthz` 超时。这会进一步放大“自动补全失效”和“登录长时间卡住”的感知问题。

## 3. 修复目标

### 3.1 功能目标

- 登录页初次打开时默认展示可直接测试的账号。
- 即使后端暂时不可用，也保留本地默认账号兜底。
- 登录请求在后端无响应时快速失败，而不是无限等待。
- 对启动阶段的瞬时失败进行一次有限重试。
- 记住最近一次成功登录的用户名，减少重复输入。

### 3.2 稳定性目标

- 不改动后端鉴权语义。
- 不引入新的基础设施依赖。
- 只在登录相关前端代码做最小侵入修改。

## 4. 实施方案

### 4.1 本地默认测试账号兜底

新增前端本地演示账号常量：

- `community_admin / 123456`
- `family01 / 123456`
- `elder01_02 / 123456`

页面初始化时先使用本地账号列表渲染快速登录区，再异步请求后端账号列表刷新。

### 4.2 最近成功账号记忆

新增本地存储键：

- `ai_health_demo_last_login_username`

行为规则：

- 登录成功后写入用户名
- 退出登录时不清除
- 下次打开登录页时优先回填最近成功账号

### 4.3 快速登录区改造

调整 `mock-accounts` 加载流程：

1. 首屏同步显示本地账号
2. 后台异步请求服务端账号
3. 请求失败时保留本地账号，不清空界面
4. 对账号列表请求做有限重试

### 4.4 登录请求超时与重试

对以下接口增加显式超时控制：

- `GET /api/v1/auth/mock-accounts`
- `POST /api/v1/auth/login`
- `GET /api/v1/auth/me`

策略如下：

- `mock-accounts` 超时：4 秒
- `login` 超时：6 秒
- `me` 超时：3 秒
- `login` 在 `408/429/500/502/503/504` 时仅重试 1 次

### 4.5 错误提示优化

错误文案区分为两类：

- 账号类：用户名或密码错误
- 服务类：后端未启动、请求超时、演示账号列表未刷新

这样用户能更快判断是该重试、该等后端、还是该检查账号。

## 5. 变更清单

### 前端

- `frontend/vue-dashboard/src/constants/demoAuth.ts`
  - 新增本地默认账号与最近登录用户名工具函数
- `frontend/vue-dashboard/src/views/LoginPage.vue`
  - 先显示本地账号
  - 异步刷新服务端账号
  - 失败时保留兜底账号
- `frontend/vue-dashboard/src/composables/useSessionAuth.ts`
  - 初始化默认用户名和密码
  - 登录成功记忆最近用户名
  - 登录与会话恢复增加超时控制
  - 登录失败时做一次轻量重试
- `frontend/vue-dashboard/src/api/client.ts`
  - 为通用 JSON 请求增加超时能力
  - 统一网络超时和服务不可达错误归因

## 6. 测试计划

### 6.1 静态验证

执行：

- `npm.cmd run typecheck`
- `npm.cmd run lint`
- `npm.cmd run build`

通过标准：

- `typecheck` 和 `build` 必须通过
- `lint` 若失败，需要确认是否为本次改动引入

### 6.2 后端接口冒烟

验证链路：

1. `GET /api/v1/auth/mock-accounts`
2. `POST /api/v1/auth/login`
3. `GET /api/v1/auth/me`

记录项：

- 是否成功返回
- 是否在超时时间内完成
- 登录用户是否正确

### 6.3 前端登录页冒烟

场景 A：后端正常

- 打开 `http://localhost:5173/`
- 检查用户名是否自动回填
- 点击登录
- 确认秒级进入系统

场景 B：后端暂时未 ready

- 打开前端页面
- 检查本地默认账号是否仍然可见
- 检查页面不会无限卡死
- 错误提示是否清晰

场景 C：错误密码

- 输入错误密码
- 确认快速失败并展示明确提示

场景 D：最近登录账号记忆

- 成功登录一次
- 刷新页面或退出后返回登录页
- 确认最近账号优先回填

## 7. 验收标准

满足以下条件即可验收：

1. 登录页默认回填测试账号
2. 后端正常时登录速度恢复到秒级
3. 后端异常时不再无限等待
4. 演示账号接口失败时仍能使用本地兜底账号
5. 最近成功登录用户名可被记忆
6. 本次改动未破坏构建和类型检查

## 8. 回滚策略

若修复引入异常，按以下顺序回滚：

1. 先回滚登录请求超时与重试逻辑
2. 保留本地默认账号兜底
3. 如仍异常，再回滚登录页相关改动

优先保留“本地默认账号回填”，因为它收益高、风险最低。

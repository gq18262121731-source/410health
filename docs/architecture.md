# 系统架构

项目采用 AIoT 微服务化思路，但为了比赛现场可控、易演示，当前仓库将核心能力先收拢为一个可运行的 FastAPI 主进程：

1. IoT 采集层：支持 BLE、MQTT、串口和 Mock 四种输入方式。
   - 当前后端可配置运行模式为 `mock`、`serial`、`mqtt`。
   - MQTT 网关模式按瑞迪莱网关原始报文接入：`data_type + BLE MAC + RSSI + BLE payload`。
2. 解析层：`iot/parser.py` 根据 T10 手环双包规则完成 `0x1803 / 0x0318` 合并。
3. 实时层：采样进入内存流服务后，立即触发规则引擎和动态 Z-Score 检测。
4. 智能层：`ai/anomaly_detector.py` 保留 LSTM AutoEncoder 接口，可在本地或 GPU 环境进一步训练。
5. 应用层：FastAPI 提供 REST API、WebSocket、AI 助手接口。
6. 展示层：Vue 看板 + Flutter 移动端骨架。

比赛演示建议：

- 先开后端 Mock 模式，自动生成 10 台设备实时数据。
- 再打开前端大屏，展示趋势图、告警、AI 建议。
- 最后切换到移动端模拟子女端或社区端入口。

## 第二阶段设备管理口径

当前设备域的推荐理解如下：

- `POST /api/v1/devices/register`
  - 只负责录入设备主档。
  - 默认返回 `bind_status=unbound`。
- `POST /api/v1/devices/bind`
  - 只允许绑定到已存在用户。
  - 默认只允许绑定到 `elder`。
- `POST /api/v1/devices/unbind`
  - 解除设备与老人归属。
- `POST /api/v1/devices/rebind`
  - 换绑到另一位老人，并保留历史。
- `GET /api/v1/devices/{mac_address}/bind-logs`
  - 查询设备绑定履历。

状态上需要区分两个维度：

- 在线状态：`online / offline / warning`
- 业务绑定状态：`unbound / bound / disabled`

不要继续把在线状态和绑定状态混用成一个字段语义。

## 前端接入建议

Vue 看板当前建议采用“渐进接入”策略：

- 社区/管理入口先在关系台账页接入设备注册、绑定、解绑、换绑和日志查看。
- 现有社区总览与子女端页面继续消费聚合后的目录和设备列表。
- 正式注册、关系管理、设备归属操作优先从后台入口落地，不要求一次性重构所有页面。

如果需要继续推进，推荐顺序是：

1. 后台页面补设备管理入口。
2. 补正式账号注册/登录入口。
3. 最后再把 mock-only 能力收拢到演示模式。

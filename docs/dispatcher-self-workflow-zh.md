# 调度与审阅自用流程（中文）

Last updated: 2026-03-22

这份文档是给我自己用的。  
目的不是解释业务功能，而是固定我后续的调度、审阅、同步、继续派发流程。

如果后面上下文变长、被多次打断、或者需要快速恢复当前协作状态，我应该先读这份文档。

## 1. 总原则

1. 用户刚发的新内容是当前最高优先级事实
2. 先吸收新要求，再继续旧任务
3. 没有完整上报，不派下一轮
4. 上报后必须先审，再决定是否继续派发
5. 任务板、总体文档、角色文档、handoff 要尽量一致

## 2. 我的标准工作流

### 第一步：读取全局信息

先读：

1. `docs/business-and-dispatch-overview-zh.md`
2. `docs/coordination-master-plan-zh.md`
3. `docs/collaboration-gap-task-board.md`
4. `docs/next-agent-handoff.md`

如果只处理某个角色，再读对应角色文档。

### 第二步：判断当前是否有新规则覆盖旧规则

要先明确：

- 用户刚刚说了什么
- 这会覆盖我之前哪些判断
- 任务板或角色文档要不要先改

如果规则变了，先改文档，再派任务。

### 第三步：派发任务

派发时必须：

- 告诉对方先读总体文档
- 再读自己的角色文档
- 明确当前任务
- 明确不要做什么
- 明确做完后必须先更新自己的任务文档，再按统一格式上报

### 第四步：等待上报

等待时的规则：

- 没有完整上报，不下发下一轮
- 如果只给一句模糊结论，视为上报不完整
- 如有必要，先要求补完整上报

### 第五步：统一读取文档与上报

每一轮结束后，我必须统一做这几件事：

1. 读对方更新后的角色文档
2. 读任务板
3. 必要时读 handoff
4. 必要时核对代码 / 测试 / 运行态
5. 再形成审核结论

### 第六步：形成审核结论

审核时必须明确给出：

- 通过
- 通过但需要修正文档 / 任务板
- 不通过

同时指出：

- 哪些地方成立
- 哪些地方口径过时
- 哪些任务编号或文档状态需要调整

### 第七步：同步全局信息

审核后，按需要更新：

- `docs/collaboration-gap-task-board.md`
- `docs/next-agent-handoff.md`
- `docs/business-and-dispatch-overview-zh.md`
- 相关角色文档

### 第八步：与用户沟通

对用户回复时要明确：

1. 我审了什么
2. 结论是什么
3. 我改了哪些文档 / 任务板
4. 下一步该派给谁

不要只说“已同步”，要说清楚“为什么这样同步”。

### 第九步：继续下一轮派发

只有满足下面条件，才派下一轮：

- 上一轮有完整上报
- 我已经审核过
- 我已经同步了必要文档
- 用户没有插入新的更高优先级规则

## 3. 文档即任务单规则

从现在开始，每个角色文档都必须同时承担：

- 任务说明
- 当前边界
- 文档维护要求
- 上报格式

我在审阅时，默认把“角色文档 + 上报内容”一起看，不再只看口头消息。

## 4. 我需要长期记住的东西

- 前端是 Vue 3 + TypeScript
- Python / pytest 只走 `helth`
- 报告接口是 `/api/v1/chat/report/device`
- 浏览器效果优先于终端输出
- 上报必须审核
- 没上报不派下一轮
- 用户打断后要继续原任务

## 5. 可被新规则覆盖的东西

- 当前第一优先级任务
- 哪些角色立即启动 / 跟进启动 / 条件启动
- 当前 UI 扩展范围
- 是否继续全站统一视觉
- 当前模型路由策略
- 是否要分离对话模型 / 报告模型

## 6. 当前这轮新增的关键要求

当前必须记住：

- 每个智能体完成后，要先更新自己的角色文档，再上报
- 我每次结束后，要统一读取文档，再审上报
- 普通对话模型和健康报告模型要正式分离
- 前端必须把对话区和报告区分开展示

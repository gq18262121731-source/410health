## 项目基线

先明确本项目当前已经具备的核心能力，后面的逐篇对比都以这条基线为准。

- `AIoT 接入链路`：支持 BLE、MQTT、串口、Mock，多源数据可以统一进后端。
- `健康监测链路`：已覆盖心率、血氧、血压、体温、跌倒状态、数据质量，并有规则引擎、健康评分、风险分级、异常解释。
- `跌倒检测链路`：已经不是单一 YOLO 检测，而是 `YOLO Pose + GRU + Hybrid TCN/Transformer + posture risk + optional detector` 的多分支融合。
- `事件化能力`：有实时状态机、事件日志、告警等级、伤情分级、前后端联动接口。
- `系统化能力`：已有后端、Web 看板、移动端骨架、设备注册/绑定/解绑、用户关系管理。
- `扩展方向`：已规划目标老人识别、target-only fall gate、多模态复核、Agent 接入。

一句话判断：你们项目已经不是“单篇论文里的一个算法或一个硬件小系统”，而是朝“可运行的智慧康养平台”走了。

- 关键摘要：提出 `DSC-OpenPose`，通过 DenseNet 式稠密连接和坐标注意力提高姿态估计，再用外椭圆参数、头部高度、下肢高度做跌倒判定；在 RF 数据集上准确率约 `98.2%`，速度约 `20.2 FPS`。
- 可参考点：
  - 适合借其 `姿态估计模型轻量化` 的思路。
  - 人体几何规则可用于增强解释性和抗误报。
  - 适合多人密集场景或公共场所跌倒监测的论述。
- 和我们项目对比：
  - 我们更强：你们已经不只是姿态估计，还加入时序模型和告警业务链。

## 总体判断

### 我们已经做得比大多数文献更好的地方

- `系统完整度更高`
  - 多数论文只做算法、硬件端或单一云平台。
  - 你们已经把 `设备接入 + 后端评分 + 摄像头监测 + 告警 + 前端展示 + 用户关系管理` 串起来了。

- `跌倒检测不是单一模型`
  - 多数文献停留在单个 YOLO 或单个姿态模型。
  - 你们已经是 `Pose + Temporal + Posture + Detector + State Machine + Injury Watch`。

- `业务化更强`
  - 多数文献输出的是精度、mAP、识别结果。
  - 你们已经有 `事件日志、告警等级、恢复观察、系统 API、页面接入`。

- `多模态潜力更强`
  - 文献中多模态常常只是“概念”或“双传感器”。
  - 你们已经有 `生命体征 + 摄像头 + 用户信息 + 设备管理 + 未来 LLM 接入` 的基础。

### 我们仍然需要重点改进的地方

- `真实场景评测还不够体系化`
  - 现在最需要补的是 `私有场景 benchmark`、`误报率/漏报率`、`检测延迟`、`每小时误报数`。

- `视觉跌倒链路的稳定性仍需提升`
  - 当前自检报告已经暴露出机器人误检、边缘碎片误检、告警入库不稳定等问题。
  - 这比继续盲目堆算法更紧急。

- `目标老人识别仍在规划态`
  - 这恰恰是你们最有差异化潜力的方向。
  - 一旦做好 `target-only fall detection`，你们会明显强于大多数同类论文。

- `事前预警能力偏弱`
  - 现阶段更像“异常发生后发现”。
  - 后续要补 `步态退化、活动能力下降、生命体征持续漂移` 的长期风险预测。

- `主动健康闭环还不够`
  - 现在有监测和告警，但干预建议、依从性追踪、效果反馈、家属/社区协同还不够闭环。

- `边缘部署和工程指标还可再做扎实`
  - 建议补 `Jetson/NUC/工控机` 实测，形成 `FPS、显存、功耗、延迟、稳定性` 数据。

- `老人端可用性和解释性需要加强`
  - 比赛时不仅要看“能不能跑”，也看“老人敢不敢用、家属看不看得懂、社区能不能接”。

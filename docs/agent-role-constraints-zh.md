# 智能体角色硬约束（中文）

Last updated: 2026-03-22

这份文档是当前项目的硬约束说明。  
如果口头理解、旧文档、终端输出和这份文档冲突，以这份文档和用户最新明确指令为准。

## 1. 技术栈约束

- 当前前端项目是 **Vue 3 + TypeScript + Vite**
- 不是 React
- 不允许按 React、React Router、`useState`、`useContext` 那套思路输出实现

## 2. 环境约束

- 绝对不要混环境
- Python / pytest 只能使用：
  - `conda run -n helth python ...`
  - `conda run -n helth pytest ...`
- 禁止使用裸 `python`
- 禁止使用裸 `pytest`

## 3. 协作约束

- 不要只分析，必须执行、验证、再上报
- 没有完整上报，不派下一轮任务
- 上报后必须先审核，再决定是否继续派发
- 如果用户中途打断，处理完插入事项后要继续原任务，除非用户明确取消
- 浏览器真实效果优先于 PowerShell 输出
- 不要把 PowerShell 乱码直接当成源码损坏
- 如发现文档、代码、运行态、任务板不一致，必须明确指出

## 4. 前端与 UI 当前阶段约束

当前阶段目标已经从“只修登录页”升级为：

- **以登录页为视觉基线**
- **向全站扩展统一视觉逻辑**

因此现在允许继续推进：

- 登录页
- 登录/注册链路拆分
- 家属首页
- 社区总览
- 成员与设备
- 告警页
- 报告页
- 趋势页

但必须遵守下面这些规则：

- 不允许每个页面重新发明一套视觉体系
- 登录页的颜色、卡片、按钮、状态标签、背景语言，是全站统一基线
- 不要为了追求页面好看而擅自改业务流程
- UI 可以推进全站统一，但不能再发明独立业务逻辑

## 5. 当前视觉基线

- 绿色主基调
- 白色 / 淡绿色 / 淡蓝色作为层级色
- 轻玻璃拟态卡片
- 安静、医疗、科技、清爽，不做重型炫技
- 中文优先、低理解成本

### 登录卡片尺寸基线

- 推荐宽度：`600px`
- 可接受宽度范围：`520px - 680px`
- 推荐最小高度：`360px`
- 可接受高度范围：`320px - 420px`
- 推荐圆角：`24px`
- 推荐内边距：`36px`
- 输入框 / 按钮高度：`48px - 52px`

原则：

- 不要做成“窄而高的表单柱”
- 要做成“居中的横向长方形面板”
- 宽高比例接近 `5:3` 或 `8:5`

## 6. 登录/注册链路约束

当前登录/注册链固定为：

1. 登录页
2. 身份选择
3. 注册（只做账号）
4. 资料完善
5. 进入系统 / 完成态

实现要求：

- 这条链应拆成小页面 / 步骤页
- 通过交互跳转推进，不要继续堆在一个超级页面里
- 登录状态源只能保留一份
- 登录相关逻辑不要在 `App.vue` 和 `LoginPage.vue` 中各持一套

## 7. 接口与协议约束

- 不允许伪造后端不存在的能力
- 报告页只能走：
  - `POST /api/v1/chat/report/device`
- 不允许再用：
  - `/chat/analyze`
  - `/chat/analyze/device`
  伪装报告

## 8. 模型展示约束

当前真实情况：

- 普通对话与健康报告还没有分成两套独立模型配置
- 它们当前仍走同一个本地模型路由，只是 prompt / query 不同

新的产品要求：

- **前端必须把“对话”和“报告”分成不同入口、不同展示区**
- 不允许继续把普通问答和正式报告混在同一个展示容器里
- 后续如果模型工程侧真正分出独立模型配置，需要再同步 FE / AG / TE

## 9. 截图与验收约束

桌面截图：

- 推荐按接近 `16:9` 的比例输出
- 推荐尺寸：
  - `1600x900`
  - `1440x810`
  - `1920x1080`

移动端截图：

- 单独归类
- 不与桌面图混用

验收顺序：

1. 先看浏览器真实效果
2. 再看截图
3. 最后才参考终端输出

## 10. 各角色边界

### Frontend Engineer

- 负责真实页面实现、状态收口、组件拆分、接口接线
- 可以改登录页、登录/注册链、业务页统一视觉落地
- 不要擅自改后端协议

### UI Designer

- 负责统一视觉逻辑、页面层级、卡片 / 按钮 / 表单 / 图表语言
- 不改业务逻辑、不改接口

### Backend Engineer

- 负责契约护航、后端联调支撑、数据库 / 网关专项
- 不为前端视觉问题随意改公共 schema

### Test Engineer

- 负责高信号回归、浏览器验收口径、演示结论
- 不要拿过期测试强卡新结构

### Health Model / Agent / Offline Model / Device Registration

- 以待命或按需介入为主
- 不抢前端统一风格与页面拆分主资源

## 11. 统一上报格式

每个智能体必须按下面格式上报：

1. 完成情况
2. 关键代码改动
3. 实际执行命令与结果
4. 当前风险 / 阻塞
5. 是否需要通知其他角色
6. 任务板应如何调整

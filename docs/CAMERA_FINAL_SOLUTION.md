## 技术说明

### 为什么不能使用RTSP？

这个摄像头使用**P2P（点对点）协议**：
- 优点：穿透NAT，远程访问方便
- 缺点：不兼容标准RTSP客户端

**P2P摄像头的特点**：
1. 通过云服务器中转
2. 使用专有协议（非RTSP）
3. 需要UID和密码配对
4. 只能通过厂商APP或SDK访问

### 快照模式的工作原理

```
摄像头 → HTTP快照请求 → JPEG图片 → 后端缓存 → WebSocket → 前端显示
  (0.3fps)              (定期抓取)      (中继)      (实时推送)   (视频流)
```

**限制因素**：
- 摄像头快照生成速度（0.3fps）
- HTTP请求开销
- 网络延迟

**优化效果**：
- 减少超时等待
- 提高抓取频率
- 优化缓存策略

## 常见问题

### Q1: 为什么FPS这么低？

**答**：这是P2P摄像头的限制。它的快照接口只能提供0.3fps，不是系统问题。

### Q2: 能否达到15fps以上？

**答**：使用当前摄像头无法达到。需要：
1. 使用摄像头专用APP（可达15-30fps）
2. 集成厂商SDK（需要开发）
3. 更换为标准RTSP摄像头

### Q3: PTZ控制为什么可用？

**答**：PTZ使用ONVIF协议（端口10080），这是标准协议，所以可用。但视频流使用P2P协议，不兼容RTSP。

### Q4: 跌倒检测会受影响吗？

**答**：不会。跌倒检测使用后端的快照流，0.3-3fps足够检测跌倒事件。

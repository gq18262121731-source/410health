# 蓝牙手环数据采集与清洗流程说明（中文）

Last updated: 2026-03-22

## 1. 这份文档是给谁用的

这份文档是给需要接手“蓝牙手环 + 蓝牙采集器”数据采集的人看的。

目标不是教人做设备工程师，而是让对方快速明白：

- 采什么数据
- 怎么采
- 采到的原始数据长什么样
- 怎么解析
- 怎么清洗
- 怎么做质检
- 最终整理成什么结构再交给后续算法、报表或业务系统

本说明当前只覆盖：

- 蓝牙手环
- 蓝牙采集器
- 串口 / 本地采集链路
- 原始数据清洗与标准化

本说明当前不覆盖：

- HTTP 网关
- MQTT 网关
- 真实线上平台接入

## 2. 当前采集链路的整体思路

建议把整条链路理解成 6 个步骤：

1. 手环广播或回应原始 BLE 数据
2. 蓝牙采集器收到原始数据并通过串口输出
3. 采集程序从串口读取文本行
4. 从文本行中提取：
   - 设备 MAC
   - 原始 payload
5. 解析程序把 payload 转成结构化健康样本
6. 对结构化样本做清洗、校验、补全、去噪，再写入下游

一句话概括：

原始串口行 -> 提取 MAC 和 payload -> 解析成健康样本 -> 清洗 -> 入库/建模/展示

## 3. 需要准备什么

### 3.1 硬件

- 蓝牙手环
- 蓝牙采集器
- USB 转串口连接线
- Windows 电脑

### 3.2 软件环境

- Python
- 串口库 `pyserial`
- 你自己的采集脚本或当前项目里的串口读取逻辑

### 3.3 采集前要确认的设备信息

- 采集器串口号
- 波特率，当前项目默认是 `115200`
- 手环 MAC 地址
- 手环输出的是哪种数据：
  - 广播包
  - 响应包 A
  - 响应包 B
  - 或两类响应组合

## 5. 原始数据是怎么来的

当前项目中，蓝牙采集器走的是串口读取方式，可以参考：
[serial_reader.py](/D:/ai_helth-main/ai_helth-main/iot/serial_reader.py)

采集器通常会不断输出文本行，每一行可能包含：

- 设备 MAC
- RSSI
- BLE 原始 payload
- 或若干控制回显信息

注意：

- 不是每一行都是真实数据
- 有些行只是：
  - `AT+...`
  - `OK`
  - `ERROR`
- 这些都属于控制信息，必须先剔除

## 6. 数据采集的标准流程

### 6.1 第一步：连接采集器

操作要点：

- 找到采集器串口
- 确认波特率
- 打开串口
- 保持稳定读取

推荐参数：

- 波特率：`115200`
- 超时：`1s`

### 6.2 第二步：初始化采集器

如果采集器支持 AT 指令，通常需要先做初始化。

当前项目的典型初始化流程是：

1. 停止扫描
2. 关闭不必要的 UUID 输出
3. 根据需要设置 MAC 过滤
4. 根据需要设置包类型
5. 开始扫描

目的：

- 减少无关输出
- 控制采集模式
- 让串口输出更稳定、更可解析

### 6.3 第三步：逐行读取串口输出

读取后，不要直接入库，先做基础筛选：

- 空行丢弃
- `AT+...` 丢弃
- `OK` / `ERROR` 丢弃
- 长度太短、明显不是十六进制数据的内容丢弃

这一层是“原始噪声过滤”。

### 6.4 第四步：从每一行里提取有效 payload

这一层的目标是从混杂文本中分离出：

- `device_mac`
- `payload`

当前项目的处理思路是：

- 先尝试识别采集器前缀格式
- 如果不是前缀格式，再尝试在整行里寻找嵌入式 payload
- 如果能识别到 MAC，就格式化成：
  - `AA:BB:CC:DD:EE:FF`

如果一行里提不出有效 payload，就不要往下走。

### 6.5 第五步：识别包类型

当前解析器会先判断 payload 属于哪一类：

- 广播包 `broadcast`
- 响应包 A `response_a`
- 响应包 B `response_b`
- 旧式响应包 `legacy_response_a / legacy_response_b`

参考解析逻辑：
[parser.py](/D:/ai_helth-main/ai_helth-main/iot/parser.py)

这一步很重要，因为不同包型里字段位置不同。

### 6.6 第六步：解析为结构化样本

解析后的样本一般至少要能拿到：

- 心率
- 体温
- 血氧

在更完整的响应里，还可能拿到：

- 电量
- 步数
- 环境温度
- 表面温度
- 血压
- SOS 标志
- UUID

### 6.7 第七步：对样本做清洗

这一步是重点，不能把解析出来的东西原样全吃掉。

## 7. 数据清洗怎么做

建议把清洗分成 5 层。

### 7.1 格式清洗

目标：

- 保证字段类型一致
- 保证 MAC、时间、数值格式统一

规则建议：

- `device_mac` 统一大写
- `device_mac` 统一为冒号分隔格式
- `timestamp` 统一为带时区时间
- 所有十六进制字符串统一去空格、去分隔符、转大写

### 7.2 基础合法性校验

目标：

- 把明显无效的数据挡掉

建议规则：

- 心率必须在合理范围内
  - 可参考：`0 ~ 240`
- 血氧必须在合理范围内
  - 可参考：`0 ~ 100`
- 体温必须在合理范围内
  - 可参考：`30.0 ~ 45.0`
- 电量必须在合理范围内
  - 可参考：`0 ~ 100`

如果超出硬边界：

- 不要直接当有效健康数据入库
- 先标记为异常样本，进入人工复核或丢弃策略

### 7.3 业务清洗

目标：

- 处理“能解析，但业务上不可靠”的数据

例如：

- 只有广播包，没有响应包时，血压、电量、步数可能不完整
- 某些温度值来自表面温度，不一定等于体温
- 同一设备短时间内高频重复上报完全相同的值，可能是重复采集

建议策略：

- 区分不同 `packet_type`
- 按包型决定字段可信度
- 不完整字段允许为空，不要强行伪造

### 7.4 多包合并清洗

有些数据不是单包就能得到完整结果，而是需要 A/B 包合并。

当前项目中：

- 响应 A 可能给出：
  - 环境温度
  - 表面温度
  - 电量
  - 步数
- 响应 B 可能给出：
  - 血压
  - 更接近体温的值

因此建议：

- 为每台设备维护一个短时缓存
- 在一个较短时间窗内尝试合并 A/B 包
- 如果超时还没等到对应包，则输出当前可用结果

当前项目里的参考思路是“部分包缓存 + 超时合并”。

### 7.5 去重和去抖

目标：

- 降低重复样本和串口噪声对数据质量的影响

建议去重键：

- `device_mac`
- `packet_type`
- 原始 payload hash
- 时间窗口

建议规则：

- 完全相同 payload 在极短时间内重复出现，可视为重复采样
- 同一设备在极短时间内出现多条完全一致样本，只保留一条
- 如果数值抖动超出物理常识，再进入异常判定，不直接用于建模

## 8. 建议的清洗后输出层级

建议把清洗后的数据分成 3 层保存。

### 第 1 层：原始层

保存原始证据，便于排错。

至少保留：

- 原始串口行
- 抽取后的原始 payload
- 设备 MAC
- 采集时间

### 第 2 层：解析层

保存解析后的结构化样本。

至少保留：

- 心率
- 体温
- 血氧
- 血压
- 电量
- 步数
- 包类型
- 原始包 A / B

### 第 3 层：清洗层

这是给后续业务、报表、算法用的。

建议新增字段：

- `is_valid`
- `quality_flag`
- `cleaning_notes`
- `dedup_key`
- `merged_from_packets`

这样别人后面看数据时，能区分：

- 这是原始解析值
- 这是清洗后的可用值
- 这条数据为什么被保留/丢弃/降级

## 9. 数据质检清单

在正式开始积累数据前，建议每次采集都检查下面这些点。

### 9.1 设备层

- 手环 MAC 是否稳定
- 同一手环是否被识别成多个 MAC
- 采集器是否持续输出
- 串口是否频繁断开

### 9.2 数据层

- 是否持续能看到广播包
- 是否能看到响应 A / B
- 合并后样本是否完整
- 是否存在大量空字段

### 9.3 质量层

- 心率是否长期为 0
- 体温是否长期固定在异常值
- 血氧是否大量超界
- 同一设备是否高频重复完全相同 payload

### 9.4 时序层

- 时间戳是否单调递增
- 是否出现明显乱序
- 是否存在长时间断流

## 10. 推荐给别人执行的采集流程

如果要交给别人照着做，可以直接按下面流程执行。

### 采集流程

1. 连接蓝牙采集器到电脑串口。
2. 确认串口号和波特率。
3. 启动采集程序，初始化采集器。
4. 持续读取串口输出。
5. 过滤控制行和空行。
6. 从每一行提取 `device_mac` 和 `payload`。
7. 识别包类型。
8. 解析成结构化样本。
9. 对样本做：
   - 格式清洗
   - 合法性校验
   - 多包合并
   - 去重去抖
10. 输出为标准化数据表。
11. 再把清洗后的数据交给：
   - 实时展示
   - 报表
   - 风险评估
   - 模型训练

## 12. 实施时的注意事项

- 不要一开始就只保留清洗后的值，原始数据一定要留。
- 不要把“解析失败”和“数据真的没有”混为一谈。
- 不要在清洗时把缺失字段强行补成正常值。
- 不要跳过包型判断直接按固定格式切字段。
- 不要忽略时间窗内 A/B 包合并的问题。
- 不要在没有去重策略的情况下直接做趋势分析。

## 13. 一句话总结

这套数据采集与清洗流程的核心原则是：

先把采集器输出稳定拿到，再把原始 BLE payload 解析成结构化样本，然后做格式统一、合法性校验、多包合并、去重去噪，最后再交给业务系统或模型使用。

# 网页端摄像头卡片问题整理

## 当前可用能力

- 后端统一连接摄像头，前端只连接后端 WebSocket，不让多个前端直接连摄像头。
- 家属端摄像头卡片已经可以显示实时画面，使用 `/ws/camera` 接收后端转发的视频帧。
- 卡片显示前端实收 fps、后端源流 fps、摄像头地址、延迟和 ONVIF 云台端口等诊断信息。
- 云台支持模拟摇杆，按住连续转动，松开停止。
- 支持拉近、拉远按钮。
- 支持暂停/继续画面、刷新状态。
- 支持摄像头麦克风监听，走 `/ws/camera/audio/listen`。
- 网页麦克风已经可以采集声音并发送到后端，走 `/ws/camera/talk/web`。

## 需要优先修的问题

### 1. 文案显示存在乱码风险

当前 `CameraMonitorCard.vue` 中部分中文文案在文件读取时出现乱码，例如按钮、状态、提示语。正式演示前必须统一修复成清晰中文。

建议处理：

- 把状态文案统一成：未配置、在线、离线、连接中、实收 fps、无视频帧。
- 把操作按钮统一成：暂停画面、继续查看、刷新状态、开始监听、监听中、局域网对讲、停止对讲。
- 把错误提示写成人能理解的话，不直接暴露底层异常。

### 2. 对讲按钮容易造成误解

当前网页麦克风能到后端，但摄像头扬声器是否播放还没有打通。按钮如果直接叫“局域网对讲”，用户会以为已经能真正从摄像头外放。

建议处理：

- 正式接入前，按钮文案改成“对讲测试”或“麦克风上行测试”。
- 状态提示明确写：已发送到后端，摄像头外放待厂商 SDK 接口确认。
- 只有检测到真正可用的摄像头 TalkData 通道后，再改成“局域网对讲”。

### 3. 监听和对讲需要更清楚地区分

监听是“摄像头麦克风 -> 家属端播放”，这个已经可用。

对讲是“家属端麦克风 -> 摄像头扬声器播放”，这个还缺厂商音频写入接口。

建议处理：

- 监听按钮只负责听摄像头声音。
- 对讲按钮只负责家属端说话。
- UI 上加一句短说明：监听可用；对讲外放通道仍在厂商 SDK 验证中。

### 4. 视频延迟和帧率诊断还不够直观

目前卡片显示 fps，但用户不知道问题发生在摄像头、后端还是浏览器。

建议处理：

- 保留三个指标：摄像头源流 fps、后端转发 fps、前端实收 fps。
- 如果前端实收 fps 明显低于后端源流 fps，提示“浏览器渲染或网络转发较慢”。
- 如果后端源流 fps 本身低，提示“摄像头源流帧率偏低”。

### 5. 云台摇杆仍需优化手感

目前已经是按住连续控制，但上下键曾经反馈不够灵敏。

建议处理：

- 保持 pointerdown 立即发送连续移动。
- pointerup / pointercancel / lostpointercapture 都必须发送 stop。
- 前端停止动画可以稍微延迟 100-150ms，但后端 stop 必须立即发。
- 后端 `CAMERA_PTZ_SPEED` 和 `CAMERA_PTZ_MOVE_SECONDS` 保持可配置。

### 6. 移动端适配需要提前考虑

当前网页卡片是桌面优先，后续移动端家属端会复用设计逻辑。

建议处理：

- 视频区域优先保持 16:9 或接近 4:3，不要被按钮挤压。
- 云台摇杆不要遮挡画面，放在画面下方或底部控制区。
- 按钮至少 44px 高，方便手机触摸。
- 对讲按钮后续移动端建议做成“按住说话”，不是点击开关。

## 建议的正式版卡片结构

```text
家中实时看护
  视频画面
  左上角：在线/连接中/离线
  右上角：实收 fps

诊断信息
  摄像头源流 fps
  前端实收 fps
  局域网延迟

控制区
  暂停/继续
  刷新状态
  开始监听
  麦克风上行测试

云台区
  模拟摇杆
  拉近 / 拉远

提示区
  错误提示
  监听/对讲状态说明
```

## 推荐优先级

1. 修复网页端卡片中文乱码和误导性文案。
2. 把“局域网对讲”临时改成“麦克风上行测试”，避免误以为摄像头已经能外放。
3. 保留监听功能，并把监听状态做清楚。
4. 优化云台摇杆触摸手感。
5. 整理卡片布局，为移动端 Flutter 家属端复刻做准备。
6. 等厂商 SDK 或 Android 原生层打通后，再把“麦克风上行测试”升级成真正“按住说话”。

## 2026-05-01 网页端收口结果

- 网页端卡片已删除对讲按钮，不再展示“局域网对讲”等容易误导用户的入口。
- 卡片中文文案已修复，状态、按钮、错误提示和说明语都改为清晰中文。
- 保留“开始监听 / 监听中”功能，明确它只代表摄像头麦克风声音传到家属端。
- 云台摇杆改成按住和拖动都能控制方向，松手立即发送停止。
- 诊断信息保留前端 fps、后端源流 fps、延迟和 ONVIF 云台端口。
- 标题说明已明确：网页端负责查看、监听和云台；远程通话后续在移动端家属端单独接入。
- 诊断信息默认折叠，只保留关键状态，避免普通家属看到过多技术指标。
- 监听开启后会在画面角落显示状态，让用户知道正在播放摄像头周围声音。
- 页面失焦或切到后台时会主动停止云台，降低误触后持续转动的风险。

## 仍需关注的问题

开发视角：

- `ScriptProcessorNode` 后续可升级为 `AudioWorklet`，减少监听音频播放抖动。
- 视频仍是后端转发 JPEG 帧，能满足演示和局域网看护，但不是最终低延迟 WebRTC 方案。
- 云台连续控制依赖后端及时发送 stop，后续可以增加后端自动保护，防止异常断连导致云台持续转动。
- 前端 API 里还保留了对讲相关方法，当前卡片不再调用；移动端远程通话方案确定后再统一清理或迁移。

用户视角：

- 家属看到“监听中”时，应明确知道是在听摄像头周围声音，而不是已经和老人通话。
- “远程通话”不要放在网页端卡片里，避免用户误点后期待摄像头外放。
- 如果画面卡顿，用户需要看到简单原因，不应该只看到技术错误。
- 云台控制要避免遮挡画面，尤其移动端复刻时摇杆应放在画面下方。
- 监听涉及隐私，正式版本建议增加明显的开启状态和关闭按钮。
- 老人侧最好有可感知提示，例如移动端真正通话时再考虑摄像头或 App 侧提示音，避免突然出声造成惊吓。

# Login/Register Structure Support

Last updated: 2026-03-22

适用阶段：登录/注册链路结构支持阶段

当前范围只覆盖 5 个小页面：

1. `Login`
2. `Role Select`
3. `Register Account`
4. `Register Profile`
5. `Register Complete`

本文件不是业务改造方案，而是给前端工程师直接落地的页面结构依据。

---

## 1. 统一视觉基线

五个页面必须共享同一套基线：

- 绿色主基调
- 白色 / 淡绿色 / 淡蓝色层级
- 轻玻璃拟态卡片
- 中文优先、低理解成本
- ECG 背景只做氛围，不抢卡片

统一原则：

- 不允许 5 个页面各自发明一套卡片样式
- 不允许某一步骤突然改成深色、大屏或运营后台风格
- 登录页是基线，后 4 个页面只做结构推进，不换视觉世界观

---

## 2. 共享背景语言

背景语言在 5 个页面中保持一致：

- 同一张静态底图
- 同一条主 ECG 线
- 同一套沿线扫光节奏
- 同一层柔和覆盖光

背景在 5 页中的角色：

- `Login`：完整展示
- `Role Select`：完全沿用
- `Register Account`：完全沿用
- `Register Profile`：完全沿用
- `Register Complete`：完全沿用，但允许扫光再弱一点

不允许：

- 某一步骤单独换背景
- 某一步骤新增满屏几何块
- 某一步骤加入粒子而其它页面没有

---

## 3. 页面结构定义

### A. Login

定位：

- 唯一明确的进入系统入口
- 卡片必须是视觉中心

卡片结构：

1. 标题区
   - 标题：`登录账户`
   - 一句说明：这是统一登录入口
2. 输入区
   - 账号输入框
   - 密码输入框
   - 错误横幅 / 回填成功横幅
3. 主按钮区
   - `登录`
4. 辅助链接区
   - 登录说明
   - 忘记密码怎么办
5. 注册入口区
   - `立即注册并完成资料引导`

要求：

- 登录卡片不承担资料完善内容
- 注册入口必须明显，但不能高于主登录按钮

### B. Role Select

定位：

- 注册链路的第一个小页面
- 先确认身份，再进入账号创建

卡片结构：

1. 标题区
   - 标题：`先选择当前注册身份`
   - 说明：先确定注册对象
2. 身份选择区
   - 老人
   - 家属
   - 社区工作人员
3. 主按钮区
   - `进入注册`
4. 返回区
   - `返回登录`

要求：

- 身份卡是主要交互，不是辅助说明
- 选中态必须足够明确

### C. Register Account

定位：

- 只负责创建账号
- 不混入资料完善和业务管理

卡片结构：

1. 标题区
   - 标题：`注册账号`
   - 说明：当前角色的账号提示
2. 输入区
   - 姓名
   - 手机号
   - 登录账号（按角色是否展示）
   - 设置密码
   - 确认密码
3. 主按钮区
   - `进入资料完善`
4. 返回区
   - `返回身份选择`

要求：

- Step 2 只做账号创建，不要把资料字段塞进来
- 错误提示必须在卡片内部靠近操作区

### D. Register Profile

定位：

- 账号创建之后的补充资料页
- 仅补充进入系统前需要展示的信息

卡片结构：

1. 标题区
   - 标题：`完善老人资料 / 家属资料 / 值守资料`
   - 一句说明：为什么要补这些信息
2. 输入区
   - 按角色展示资料字段
3. 补充选项区
   - bind now / later
   - 首屏查看偏好
4. 主按钮区
   - `完成资料并创建账号`
5. 返回区
   - `返回账号创建`

要求：

- 不新增设备绑定和关系绑定
- 只承接已有流程里“资料完善”的表达

### E. Register Complete

定位：

- 完成态，不是瞬时 toast
- 承接“注册结束 -> 回到登录 -> 回填账号”

卡片结构：

1. 标题区
   - 标题：`注册完成，准备回到登录`
   - 成功横幅
2. 结果摘要区
   - 登录账号
   - 下一步去向
3. 信息确认区
   - 当前资料摘要
4. 主按钮区
   - `回到登录并自动回填`
5. 返回区
   - `返回资料完善`

要求：

- 完成态必须明确告诉用户“点击后会自动回填账号”
- 不允许注册成功后自动关闭而不展示完成态

---

## 4. 步骤条统一规则

步骤条固定为 4 步：

1. 身份选择
2. 注册账号
3. 资料完善
4. 完成回填

说明：

- 登录页本身不属于这 4 步步骤条
- 登录页是链路入口，注册链路从 Role Select 开始

视觉要求：

- 当前步骤必须高亮
- 已完成步骤和当前步骤要有明显区别
- 小屏下允许单列堆叠

---

## 5. 前端落地建议

当前建议继续沿用：

- `frontend/vue-dashboard/src/views/LoginPage.vue`
- `frontend/vue-dashboard/src/components/PublicRegistrationPanel.vue`
- `frontend/vue-dashboard/src/demo-theme.css`

落地方式：

- `LoginPage.vue` 负责：
  - Login 页面
  - 与注册面板的切换
  - 背景层
  - 登录卡片结构
- `PublicRegistrationPanel.vue` 负责：
  - Role Select
  - Register Account
  - Register Profile
  - Register Complete
- `demo-theme.css` 负责：
  - 统一卡片、按钮、输入框、状态标签、步骤条、背景和 ECG 语言

建议复用块：

- `login-card-section`
- `register-stage-card`
- `register-progress`
- `feedback-banner`
- `status-tag`

不建议：

- 把 4 个注册步骤拆成完全独立视觉组件再各写一套样式
- 在不同步骤里换按钮语义和位置

---

## 6. 当前风险

- 登录链路虽然已经拆成小页面结构，但仍需浏览器侧按 16:9 标准截图做最终验收。
- `LoginPage.vue` 和 `PublicRegistrationPanel.vue` 的结构已经具备 5 页表达，但后续 FE 若调整文案或字段顺序，必须保持结构顺序不变。
- 当前阶段不允许继续发散到无参考图业务页，否则会重新打散这套基线。

---

## 7. 对 FE 的直接要求

前端工程师继续落地时必须保证：

- 不改业务流程
- 不新增设备绑定
- 不新增关系绑定
- 不改接口流程
- 5 个页面共用同一背景语言
- 5 个页面共用同一按钮体系
- 5 个页面共用同一输入框和反馈横幅体系

# 姿态检测、大模型微调、Agent 与文科背景对比笔记

日期：`2026-05-10`

目标：

- 补充 `姿态检测` 方向前沿论文
- 补充 `大模型微调` 方向论文
- 补充 `智能体 / Agent` 方向论文
- 补充 `文科背景 / 养老痛点` 方向论文
- 明确这些方向下，你们项目当前的不足

## 先看项目现状

基于仓库现状，先做一个硬判断：

### 1. 姿态检测现状

- 当前在线跌倒主链路的姿态前端仍是 `yolo11n-pose`
- 已有多分支时序融合：
  - `GRU temporal`
  - `Hybrid TCN + Transformer`
  - `posture risk classifier`
  - `optional detector branch`
- `RTMPose / RTMO` 在文档里已经明确被视为下一步方向，但尚未接入主链路
- `ByteTrack / OSNet / ArcFace / AdaFace / OpenGait` 都还在规划里，没有真正落地

一句话：你们“会用姿态”，但还没有进入“前沿姿态前端 + 多目标稳定跟踪 + 身份关联”阶段。

### 2. 大模型微调现状

- 后端已经有 `Qwen / Ollama` 接入
- 已有 `AgentModelSuite`、`RAG`、`ToolAdapter`
- 已有 `fall multimodal review`、`analysis report` 等能力
- 但仓库内没有看到完整的：
  - `LoRA / QLoRA` 微调脚本
  - `SFT 数据集构建流程`
  - `偏医疗/养老领域的专用指令微调链路`
  - `评测基准与对照实验`

一句话：你们已经“在调用大模型”，但还没有“把大模型变成你们自己的养老专用模型”。

### 3. Agent 现状

- 已有 `family agent`、`community agent`
- 已有工具调用：
  - 实时样本
  - 趋势
  - 设备状态
  - 绑定历史
  - 老人档案
  - 告警
  - 社区概览
- 已有 `RAG + tools + response normalization`
- 已有一定程度的工作流约束

但仍然缺：

- 多智能体协同决策机制
- 任务拆分与角色协商
- 工具可靠性评估与回退策略体系化
- 真正的临床/照护场景流程仿真评测

一句话：你们已经有“Agent 功能块”，但还不是严格意义上经过系统评测的“医疗/养老多智能体系统”。

## 一、姿态检测方向

### 推荐关注文献

#### 1. RTMO: Towards High-Performance One-Stage Real-Time Multi-Person Pose Estimation

- 来源：[CVPR 2024](https://openaccess.thecvf.com/content/CVPR2024/html/Lu_RTMO_Towards_High-Performance_One-Stage_Real-Time_Multi-Person_Pose_Estimation_CVPR_2024_paper.html)
- 关键点：
  - 一阶段多人姿态估计
  - 主打 `实时性 + 精度平衡`
  - 更适合边缘部署和多人场景
- 对我们项目的启发：
  - 你们现在 `yolo11n-pose` 能跑，但不是专门为多人姿态估计最优设计
  - 如果要做 `目标老人识别 + 多人环境下只跟目标老人`，姿态前端需要更稳

#### 2. ByteTrack: Multi-Object Tracking by Associating Every Detection Box

- 来源：[ECCV 2022](https://www.ecva.net/papers/eccv_2022/papers_ECCV/html/315_ECCV_2022_paper.php)
- 关键点：
  - 核心不是“只保高分框”，而是把低分框也纳入跟踪关联
  - 对遮挡、轨迹断裂、多目标 ID 稳定很关键
- 对我们项目的启发：
  - 你们 target-user 方向最缺的不是单帧识别，而是 `持续跟踪 + 不串人`
  - 当前没有真正稳定的 `track id -> target user -> fall chain`

#### 3. AdaFace: Quality Adaptive Margin for Face Recognition

- 来源：[arXiv 2022](https://arxiv.org/abs/2204.00964)
- 关键点：
  - 低质量、遮挡、模糊脸图下更稳
  - 用质量感知 margin 改善识别
- 对我们项目的启发：
  - 你们规划里已经写到 `AdaFace`
  - 很适合居家摄像头这种侧脸、糊脸、逆光脸场景

#### 4. OpenPose 系列与多人姿态估计基础

- 代表文献：[Realtime Multi-Person 2D Pose Estimation Using Part Affinity Fields](https://openaccess.thecvf.com/content_cvpr_2017/html/Cao_Realtime_Multi-Person_2D_CVPR_2017_paper.html)
- 作用：
  - 不是最新，但适合给答辩时说明“多人姿态估计”的经典路线
  - 便于和你们现有 YOLO pose 路线做技术比较

### 这一方向下我们项目的不足

#### 1. 姿态前端还不够前沿

- 当前主前端仍是 `yolo11n-pose`
- 文档里已经承认 `RTMPose / RTMO` 是 promising next-step
- 说明你们自己也知道，现有姿态前端不是最终解

#### 2. 缺稳定多人跟踪

- 没有真正落地 `ByteTrack / BoT-SORT + identity persistence`
- 这会直接影响：
  - 多人场景误跟踪
  - target-only 检测失效
  - 跌倒事件串人

#### 3. 缺身份识别与姿态的统一融合

- 你们现在有目标老人方案规划
- 但还没有真正形成：
  - face id
  - body cue
  - track id
  - fall event
  - alert routing
  这一整条稳定链路

#### 4. 夜间 / 遮挡 / 侧背面仍是明显薄弱点

- 当前更多是日间可见光链路
- 低照、背身、遮挡、多目标贴近时，风险仍高

### 可以直接提炼成项目改进项

- 把 `RTMO / RTMPose` 作为下一代姿态前端候选，做 A/B 测试
- 引入 `ByteTrack` 做多人稳定跟踪
- 引入 `AdaFace + OSNet` 做目标老人识别
- 对 `多人 + 遮挡 + 低照 + 侧背面` 做专项评测集

## 二、大模型微调方向

### 推荐关注文献

#### 1. LoRA: Low-Rank Adaptation of Large Language Models

- 来源：[arXiv 2021](https://arxiv.org/abs/2106.09685)
- 关键点：
  - 冻结原模型，只训练低秩适配器
  - 微调成本低，推理不增加明显时延
- 对我们项目的意义：
  - 如果你们要做“养老健康专用模型”，LoRA 是最现实的起点

#### 2. QLoRA: Efficient Finetuning of Quantized LLMs

- 来源：[arXiv 2023](https://arxiv.org/abs/2305.14314)
- 关键点：
  - 4-bit 量化 + LoRA
  - 在单卡上微调更大模型
  - 很适合资源受限环境
- 对我们项目的意义：
  - 你们比赛和本地部署场景都更接近“资源有限”
  - 如果要做本地养老问答、告警解释、报告生成，QLoRA 很现实

#### 3. Towards a Personal Health Large Language Model

- 来源：[arXiv 2024](https://arxiv.org/abs/2406.06474)
- 关键点：
  - 面向个人健康数据
  - 能融合可穿戴时序数据，生成个性化健康建议
  - 不是只会医学问答，而是会解释个人健康趋势
- 对我们项目的意义：
  - 这篇和你们最贴
  - 你们已经有生命体征时序 + 风险评分 + 报告生成
  - 下一步应该是做“面向你们数据结构的专属健康模型”

### 这一方向下我们项目的不足

#### 1. 没有自己的领域微调模型

- 当前更像：
  - 通用模型 + Prompt
  - 通用模型 + 工具调用
  - 通用模型 + RAG
- 而不是：
  - 养老领域指令微调模型
  - 跌倒事件解释专用模型
  - 健康报告风格对齐模型

#### 2. 缺少训练数据工程

- 没看到系统化的：
  - 对话样本清洗
  - 报告样本构造
  - 告警解释样本
  - 家属/社区双角色样本

#### 3. 缺少评测闭环

- 没有看到标准化评测：
  - 专业性
  - 一致性
  - 幻觉率
  - 安全性
  - 养老场景适配度

#### 4. 缺“结构化数据 + 时序数据 + 文本生成”的联合微调思路

- 你们的特色数据不是纯文本
- 是：
  - 生命体征时序
  - 风险评分
  - 跌倒事件
  - 恢复状态
  - 家属/社区上下文
- 这意味着你们不应该只做普通聊天微调，而要做 `personal health reasoning` 方向

### 可以直接提炼成项目改进项

- 构建 `养老健康问答 + 告警解释 + 报告生成` 的 SFT 数据集
- 先做 `Qwen + LoRA/QLoRA` 的小规模领域适配
- 单独评测：
  - 健康报告生成
  - 跌倒事件解释
  - 家属沟通建议
  - 社区处置建议

## 三、智能体 / Agent 方向

### 推荐关注文献

#### 1. AI Hospital: Benchmarking Large Language Models in a Multi-agent Medical Interaction Simulator

- 来源：[arXiv 2024](https://arxiv.org/abs/2402.09742)
- 关键点：
  - 把医生、病人、检查者、上级医生做成多智能体模拟
  - 不只评单轮问答，而评多轮互动、信息收集、检查建议、诊断流程
- 对我们项目的意义：
  - 你们现在 Agent 主要还是“分析器”
  - 还没有进入“多角色协同 + 流程模拟 + 任务分工”阶段

#### 2. AgentClinic: a multimodal agent benchmark to evaluate AI in simulated clinical environments

- 来源：[arXiv 2024](https://arxiv.org/abs/2405.07960)
- 关键点：
  - 多模态
  - 临床环境仿真
  - 评估模型在信息不完全条件下如何调工具、做判断
- 对我们项目的意义：
  - 和你们的真实问题很像：
    - 监测信息不完整
    - 设备数据有缺失
    - 图像有噪声
    - 人工复核有限

#### 3. Evaluating large language models as agents in the clinic

- 来源：[npj Digital Medicine 2024](https://www.nature.com/articles/s41746-024-01083-y)
- 关键点：
  - 强调 LLM 做 agent 时，不能只看答题能力
  - 要看工具使用、流程执行、临床安全和稳定性
- 对我们项目的意义：
  - 非常适合拿来支撑你们“Agent 不等于聊天机器人”的论述

### 这一方向下我们项目的不足

#### 1. Agent 仍偏单体分析器，不是协同系统

- 你们已有：
  - family agent
  - community agent
- 但它们更像两个视角不同的分析器
- 还不是：
  - 告警分流 agent
  - 风险复核 agent
  - 家属沟通 agent
  - 社区处置 agent
  - 设备巡检 agent
  这种协作网络

#### 2. 缺严格的工具可靠性治理

- 已有工具调用是好事
- 但还缺：
  - 工具失败回退策略
  - 工具结果置信度
  - 多工具结果冲突处理
  - 错误传播隔离

#### 3. 缺场景化 benchmark

- 当前没有看到你们自己的：
  - 跌倒后处置流程 benchmark
  - 家属咨询 benchmark
  - 社区多设备优先级处理 benchmark

#### 4. 缺“多模态 agent”

- 当前 Agent 主要还是吃结构化数据和文本
- 图像、快照、视频证据并没有系统纳入 Agent 推理主链

### 可以直接提炼成项目改进项

- 建一个 `养老照护流程 Agent benchmark`
- 至少拆出 3 类 agent：
  - `健康解释 agent`
  - `告警处置 agent`
  - `社区优先级调度 agent`
- 对高风险跌倒事件接入 `多模态复核 agent`

## 四、文科背景、需求与痛点方向

这一部分非常重要。比赛里如果只有技术，没有社会问题和用户痛点支撑，项目会显得“能做但不一定有人需要”。

### 推荐关注文献 / 方向

#### 1. 老年慢性病患者数字健康技术焦虑现状及影响因素研究

- 来源：[老年医学研究公开页](https://yxln.cbpt.cnki.net/portal/journal/portal/client/paper/0ea52c5a97783511787b14fb273e60f6)
- 可提炼痛点：
  - 老年人对数字健康技术存在焦虑
  - 不是有设备就会用
  - 技术接受度、认知负担、隐私担忧都会影响使用

#### 2. 老年人数字健康技术焦虑的研究进展

- 来源：[PDF 预览页](https://yxln.cbpt.cnki.net//portal/journal/portal//client/paper/preview?cache=false&filePath=files%2Fyxln%2Fhtmlfulltext%2Fpdf%2F2026%2F01%2F%E8%80%81%E5%B9%B4%E4%BA%BA%E6%95%B0%E5%AD%97%E5%81%A5%E5%BA%B7%E6%8A%80%E6%9C%AF%E7%84%A6%E8%99%91%E7%9A%84%E7%A0%94%E7%A9%B6%E8%BF%9B%E5%B1%95.pdf)
- 可提炼痛点：
  - 老年人技术焦虑与认知下降、使用门槛、操作复杂度强相关
  - 适合支撑你们“低门槛交互设计”的必要性

#### 3. 浅谈社区老年人对便携式可穿戴设备的需求——以智能手环为例

- 来源：[万方医学](https://med.wanfangdata.com.cn/Paper/Detail/PeriodicalPaper_zglnbjyx202004007)
- 公开信息可提炼：
  - 只有小部分老年人真正了解智能手环
  - 老年人更关心：
    - 数据准确性
    - 操作便捷
    - 佩戴舒适
    - 一键呼救
    - 跌倒警报
    - GPS 定位
- 这篇非常适合写用户需求

#### 4. 健康老龄化视角下的老年数字鸿沟：成因与治理

- 来源线索：[陕西理工大学学报社会科学版门户](https://hzsf.cbpt.cnki.net/portal/journal/portal/client/paper/3de06abf66b693af0c760f5848d5339a)
- 可提炼痛点：
  - 数字鸿沟不是抽象问题，会直接影响健康服务可及性
  - 对农村老人、低教育程度老人影响更大

#### 5. 积极老龄化视角下智慧养老服务供给研究

- 来源：[万方学位](https://d.wanfangdata.com.cn/thesis/D01980708)
- 可提炼痛点：
  - 智慧养老供给不足
  - 服务参与度不高
  - 供需错配明显

### 这一方向下我们项目的不足

#### 1. 痛点叙事还不够细

- 现在项目文档更多偏技术路线
- 还缺：
  - 独居老人为什么真的需要它
  - 家属为什么愿意用
  - 社区为什么会接
  - 老人为什么可能拒绝用

#### 2. 缺“数字鸿沟”视角

- 你们项目很容易默认：
  - 老人会操作
  - 家里有网
  - 设备总在线
  - 家属愿意频繁看 App
- 但现实不是这样

#### 3. 缺社会学 / 公共管理维度的实施障碍分析

- 例如：
  - 老人隐私顾虑
  - 监护边界不清
  - 社区人手不足
  - 设备维护成本
  - 责任归属不清

#### 4. 缺老年友好交互证据

- 你们现在已有系统功能
- 但还没有充分证明：
  - 字体、提醒频率、交互流程、反馈形式适合老人
  - 不会引发技术焦虑

### 可以直接提炼成项目改进项

- 在汇报中单列 `独居老人、家属、社区` 三类痛点
- 单列 `数字鸿沟与技术焦虑` 作为项目必要性
- 单列 `适老化设计` 作为非技术创新点

## 五、四个方向合起来看，我们项目最缺什么

### 1. 技术上最缺

- 更先进、更稳定的多人姿态前端
- 真正可用的目标老人跟踪与识别链路
- 领域微调的大模型，而不是只靠 Prompt
- 经过评测的多智能体协作机制

### 2. 产品上最缺

- 老人端适老化交互证据
- 家属/社区的处置流程闭环
- 高风险事件的多模态复核闭环

### 3. 研究上最缺

- 自己的 benchmark
- 自己的微调数据集
- 自己的多角色流程评测
- 自己的用户需求与痛点调研

## 六、建议你们下一步怎么补

### 第一优先级

- 做 `目标老人识别 + ByteTrack + 多人姿态稳定跟踪`
- 为 `跌倒复核 / 健康解释 / 家属建议` 构建小型 SFT 数据集
- 为 Agent 建 `养老场景 benchmark`

### 第二优先级

- 评测 `RTMO / RTMPose` 是否优于现有 `yolo11n-pose`
- 做 `Qwen + LoRA/QLoRA` 领域适配
- 把图像快照纳入 Agent 推理链

### 第三优先级

- 补“文科材料”：
  - 数字鸿沟
  - 技术焦虑
  - 智慧养老供需错配
  - 老人可穿戴需求
